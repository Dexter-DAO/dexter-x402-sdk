# `@dexterai/x402` — SDK Audit

**Date:** 2026-05-20 (initial) · 2026-05-20 (revised post-sponsored-access correction) · 2026-05-20 (rebaselined for 3.8.1)
**Scope:** every public entrypoint, every flagged-suspicious file, the README, the CHANGELOG, the full usage map across 30+ sibling repos in `~/websites/` (dexter-api, dexter-fe, dexter-facilitator, x402gle, x402-ads, dexter-mcp, dexter-loop, dexter-agents, **opendexter-ide, dexter-lab, dexter-bakeoff, skillsmith-cli, dexter-decks, dexter-agents-v2, x402ads-io**, and more).
**Repo state at audit:** SDK **v3.8.1** (audit originally written against 3.7.8 — see §0a for what landed between then and now), 197 commits over ~5 months, ~12K LOC source, ~3K LOC tests, 7 public entrypoints.

## 0. Errata from initial draft

The first version of this audit (in chat, before this document existed) declared several SDK exports "dead" based on a usage sweep across **only 8 sibling repos**. Branch correctly challenged the claim about sponsored-access. Re-running the sweep across **30+ repos** changed the verdict on multiple findings:

| Original claim | Reality |
|---|---|
| `sponsored-access` "zero consumers" | **Wrong.** 4 production consumers: `opendexter-ide/packages/x402-mcp-tools/src/tools/fetch.ts`, `dexter-mcp/toolsets/x402-client/index.mjs`, `dexter-mcp/open-mcp-server.mjs`, `x402gle/src/components/instinct/InstinctReceipt.tsx`. All use `await import('@dexterai/x402/client')` dynamic imports — which a static grep for `from '@dexterai/x402'` will miss. |
| `createBudgetAccount` "zero consumers" | **Wrong.** Used by `skillsmith-cli/src/lib/wallet.ts`. |
| `payAndFetch` "1 consumer" | **Undercounted.** Real count is at least 2 source consumers (`opendexter-ide/packages/x402-mcp-tools/src/tools/fetch.ts`, `dexter-api/src/tasks/verifier/payment.ts`); dexter-bakeoff also has multiple references in test/bakeoff results documentation, suggesting active comparison testing. |
| `x402BrowserSupport`, `createDynamicPricing`, `x402AccessPass` "marketing-demo only" | **Partially wrong.** Also referenced by `dexter-lab`'s **agent code-template generator** (`app/lib/.server/agent/`, `app/utils/templates/`) — the agent emits these as suggestions in generated code. That's a "meta-consumer": removing the SDK export breaks the *advice* the agent gives, even if no app imports it directly. |

**Methodology lesson for future re-runs:**

1. Sweep **every** repo in `~/websites/`, not a hand-picked list.
2. Grep for **dynamic imports** (`await import\(['"]@dexterai/x402`) in addition to static `from '@dexterai/x402'`.
3. Distinguish four consumer classes: **production source**, **marketing demo**, **doc/MDX mention**, **agent template emission**. Each calls for a different decision.

The findings below have been revised on that basis. Items that survived the recount are still high-confidence; items that didn't are downgraded with reasoning.

## TL;DR

The SDK has a **modern, well-engineered core** (`payment/`, `batch-settlement/`, the 3.1/3.2 cleanup work) wrapped in a **discoverability problem**. The newest, best code paths (`payAndFetch`, `sponsored-access`, batch-settlement) are buried in the public surface and undersold in the README, while a handful of January-era stopgaps still dominate the headline real-estate.

Three patterns explain "not as good as competitors":

1. **The recommended path is buried.** `payAndFetch` is the cleanest, version-agnostic client API and it's listed 8th in `client/index.ts`. The README Quick Start doesn't mention it.
2. **The README sells the wrong feature set.** Big sections promote Access Pass and Dynamic Pricing (which have only marketing-demo + agent-template uses) while underselling Sponsored Access (which is wired into the MCP tools ChatGPT/Claude agents actually call) and `payAndFetch`.
3. **A genuinely useful primitive (Budget Account) is underbuilt.** 205L, in-memory only, never iterated since March. The agent-safety story competitors probably have a better version of.

**Fix:** delete one ~800-line dead file (`model-registry.ts`), `@deprecated` four more, **rewrite the README to lead with what's actually good**, and **rebuild Budget Account** as the persistent/observable primitive it deserves to be. Detail in §4, §6, §7.

## 2. What's wrong or weak

### 🔴 Critical — the SDK exports two `ModelPricing` types from the same entrypoint

`src/server/index.ts` lines 81-123 re-exports both:

- `import { ModelPricing } from './token-pricing'` (cost-per-token shape)
- `import { ModelPricing as RegistryModelPricing } from './model-registry'` (the OpenAI catalog row shape)

A consumer who writes `import { ModelPricing } from '@dexterai/x402/server'` gets the **first one**, and may not realize the same name in the model-registry section means a structurally different thing. **Naming collision in a public surface is a real bug**, not a cosmetic one.

### 🔴 Critical — `src/server/model-registry.ts` is 789 LOC of dead code

- Born 2026-01-11. Single commit. Never touched again.
- Hardcoded OpenAI pricing snapshot. The file's own doc-comment says "Updated: January 2026." **Already stale.**
- **Zero consumers** in the entire umbrella (verified by grep across all 8 sibling repos).
- 12 helpers exported, none called.
- It also collides with `token-pricing.ts`'s `ModelPricing` type.

This is dead weight that confuses every reader of the public surface. Delete it.

### 🔴 Critical — `src/server/access-pass.ts` is a v1 stopgap still flagship in the README

- 442 LOC. Born 2026-02-09.
- The README's **section 2 ("Why This SDK")** sells it as "The Stripe replacement for crypto-native APIs."
- **Zero production consumers.** The only files referencing it are:
  - `dexter-fe/app/access-pass/page.tsx` — marketing demo
  - `dexter-fe/app/access-pass/layout.tsx`
  - `dexter-fe/app/rpc/components/PassPanel.tsx`
  - `dexter-fe/app/sdk/v2/V2Page.tsx`
- All four are dexter-fe demo/marketing surfaces. The most recent dexter-fe commits touching it have subjects like "polish: clean up access pass demo page for production" — i.e., its existence is for the showcase, not because production needs it.
- Same story with `useAccessPass` in `react/`: 3 consumers, all dexter-fe demo pages.

This is the single biggest contributor to "SDK feels not as good as competitors." It dominates the README real-estate and the public surface area without serving real customers.

### 🟠 High — `payment/dispatcher.ts` lies about `network` when no payment is required

`src/payment/dispatcher.ts:104-112`:

```typescript
if (probe.status !== 402) {
  // not-applicable placeholder — a proper fix needs a types.ts change, out of Task 8 scope
  return {
    ok: true,
    response: probe,
    amountPaid: '0',
    network: { caip2: '', bare: '', family: 'evm' },
  };
}
```

If a caller uses `result.network` for analytics or routing on a successful no-payment response, they get **EVM Base by default**. The author flagged it in a comment, but it's shipped. Either:

- Change `PayResult.ok: true` to make `network` optional, or
- Add a third discriminator (`{ ok: true, paid: false, response }` vs `{ ok: true, paid: true, response, network, amountPaid, txSignature? }`)

### 🟠 High — three overlapping pricing systems with no recommendation

- `dynamic-pricing.ts` (265L, born 2025-12, last touched 2026-03) — character-based
- `token-pricing.ts` (393L) — tiktoken-based
- `model-registry.ts` (789L) — hardcoded OpenAI catalog

The README presents all three as features. A new adopter has no idea which to use. The answer is *probably* none of them — most consumers price externally; only 1 dexter-fe demo uses `createDynamicPricing` and 3 callers use `createTokenPricing`. This is symptom-of-iteration debt.

### 🟠 High — three client APIs, no recommendation hierarchy

Public surface offers three ways to fetch a paid resource:

1. `wrapFetch(fetch, { walletPrivateKey })` — Node helper, **7** consumers
2. `createX402Client({ wallets })` — **3** consumers
3. `payAndFetch(url, init, wallets, opts)` — **1** consumer, but it's the cleanest design

`payAndFetch` is the **newest, version-agnostic, properly-discriminated-union-typed entrypoint** and almost nobody knows about it. It's listed **8th** in the `client/index.ts` exports. The README doesn't mention it in its Quick Start. **The recommended path is buried under the legacy paths.**

### 🟡 Medium — `src/client/budget-account.ts` (205L)

Single-commit on 2026-03-15. **One production consumer:** `skillsmith-cli/src/lib/wallet.ts` (revised — initial audit said zero; missed this repo).

**What it does:** wraps `wrapFetch` with a spending limit, per-request cap, per-hour rate limit, and optional domain allowlist. Keeps an in-memory ledger.

**Why it's underbuilt:** ledger is in-memory (lost on restart), stuck at v0 minimum-viable, never iterated. A competitor SDK's "agent budget" feature would be persistent, observable, per-user, with envelopes and approval flows. This is a 205-line stub of a feature that wants to be 2000 lines.

**Verdict:** real product value, **has one real consumer**, current implementation is underbuilt for prod-grade agents. Don't delete — **rebuild for 4.0** as the persistent/observable agent-budget primitive. The skillsmith-cli consumer is exactly the kind of integration this primitive is for; building it out properly serves that use case directly.

### 🟢 Low — `src/client/sponsored-access.ts` (94L) — **NOT DEAD**

Single-commit on 2026-03-15. Initial audit said zero consumers; revised after Branch challenged that claim. Re-sweep across 30+ repos found **four production source consumers**, all using `await import('@dexterai/x402/client')` (which the original static grep missed):

- `opendexter-ide/packages/x402-mcp-tools/src/tools/fetch.ts:380` — the MCP `fetch` tool extracts recommendations from every paid response and fires the impression beacon.
- `dexter-mcp/toolsets/x402-client/index.mjs:247` — Dexter MCP server's x402 client tool.
- `dexter-mcp/open-mcp-server.mjs:574` — the open MCP server (used by Claude/ChatGPT MCP integrations).
- `x402gle/src/components/instinct/InstinctReceipt.tsx` — the Instinct receipt UI renders these recommendations to humans browsing settlements.

**What it does:** buyer-side helpers (`getSponsoredAccessInfo`, `getSponsoredRecommendations`, `fireImpressionBeacon`) that read sponsored recommendations off x402 payment receipts and fire an impression beacon to the Instinct ad network.

**Verdict:** **keep as-is.** This module is the buyer-side reader for the entire Instinct ad-network product, and it's wired into the two highest-visibility surfaces (MCP tools that ChatGPT/Claude/Cursor agents call, plus the human-facing receipt UI on x402gle). The Instinct ad-network thesis depends on these helpers being a first-class part of the buyer-side SDK — they should NOT be moved to a sub-package or marked deprecated.

If anything, the README should *promote* this module more clearly: today the "Sponsored Access" README section reads like an aspiration; in reality it's the connective tissue for a real product that's already shipped to MCP agents.

### 🟡 Medium — `src/server/browser-support.ts` (461L)

Renders HTML paywall pages for browser 402s. **Zero consumers anywhere.** The README mentions it implicitly in "Server SDK" but doesn't dedicate a section. If it was meant as a thing nobody-has-to-write themselves but everyone gets for free, it's not getting picked up. Either find one real use and document it, or delete it.

### 🟡 Medium — `src/react/` last touched 2026-03-15

Two months stale. `useX402Payment` has 10 consumers (real) but worth a fresh-eyes pass on whether the hook surface matches the patterns the newer `payment/` module promotes. `useAccessPass` should go (see Critical above).

### 🟡 Medium — `network-map.ts` hardcodes the chain list

`src/payment/network-map.ts:17-29` lists 8 chains with a comment that says "extend as the facilitator adds chains." Adding a chain on the facilitator requires editing this file. Single-source-of-truth problem — should pull from the same constants `adapters/evm.ts` and `constants.ts` already centralize.

### 🟡 Medium — README sells the wrong feature set

Section ordering is roughly: x402 explainer → marketplace discovery → Quick Start → batch settlement → Access Pass (large) → Manual Server → Dynamic Pricing (large) → Token Pricing (large) → Sponsored Access (large) → API Reference.

Three of the four largest sections (Access Pass, Dynamic Pricing, Sponsored Access) are features with **zero real consumers**. The **two** sections that describe what people actually use (`wrapFetch` flows, `x402Middleware`) are short and buried under headlines about Access Pass and tiktoken.

A new adopter reading top-down meets the dead features first. Reorder so the working canonical paths lead, and demote/remove the legacy showcase content.

### 🟡 Medium — `wrapFetch` is recommended for Node and `createX402Client` for browser, but they overlap

The README's Quick Start says "use `wrapFetch` in Node, `createX402Client` in browser." But `createX402Client` works in Node. The split exists because `wrapFetch` accepts only private-key strings, while `createX402Client` accepts wallet objects. That's an internal capability difference being exposed as a Node-vs-browser split.

`dexter-facilitator ^1.7.2` and `dexter-mcp ^2.0.0` are particularly stale — neither actually imports anything from the SDK in source code. The dependency could be dropped from those `package.json` files entirely.

## 5. What's missing (capabilities competitors probably have)

| Missing | Why a peer SDK likely has it |
|---|---|
| **A single canonical "happy path" example in the README** | Right now there are 5 different example shapes in the first 200 lines (`wrapFetch`, `createX402Client` browser, `createX402Client` Node, `useX402Payment`, batch-settlement). A modern SDK README opens with **one** example that handles 95% of cases. |
| **Receipts / observability hooks** | `getPaymentReceipt` exists but is mentioned once. There's no metrics hook, no structured logging interface — every internal log uses `console.log` (verbose) or `console.error`. A peer SDK would emit events you can subscribe to (`onPaymentAttempt`, `onSettlementFailed`, etc.). |
| **First-class refund / dispute path** | dexter-api + dexter-fe have `/admin/refunds` infra. The *SDK* offers nothing for the seller to programmatically issue a refund. If a customer disputes a payment, the seller drops into raw chain calls. |
| **Pluggable storage at the SDK level** | The batch-settlement seller defaults to `FileChannelStorage`. That's correct for v1, but the README doesn't show a Redis / Postgres recipe. Most prod deployers will need one. |
| **A real policy abstraction for spend limits** | `wrapFetch` has `maxAmountAtomic`. `createBudgetAccount` (now confirmed used by skillsmith-cli) adds total/per-request/per-hour caps + domain allowlist, but it's in-memory only and underbuilt. The 4.0 rebuild (§6) is the move. |
| **Typed scheme registry** | The SDK currently treats schemes (`exact`, `permit2`, `batch-settlement`) as strings inside `ChallengeOption.scheme`. A typed enum + capability discovery (`schemesForNetwork(network): SchemeId[]`) would help callers reason about what to expect. **Partially answered by 3.8.0** — the new `ResourceServerExtension` registry is the right shape for capability discovery on the *extension* axis. Doing the same for *schemes* themselves is the natural next step. |
| **End-to-end retry/backoff config** | A peer SDK competing with Stripe-flavored APIs lets you configure retry on network errors, idempotency keys, etc. Right now `payAndFetch` does one probe + one paid attempt, no retry. |
| **Streaming-friendly body support** | `payAndFetch` explicitly rejects non-string bodies at `dispatcher.ts:75-86`. A modern SDK would buffer-then-replay or accept an `AsyncIterable`. Correctness-over-convenience today; a competitor might just handle it. |

The single-commit, in-memory `createBudgetAccount` has 1 real consumer and an obvious product role (the agent-safety differentiator). 4.0 rebuild scope:

- **Observability hooks** (`onSpend`, `onLimitApproached`, `onBlocked`).

- **Persistent ledger** so restart doesn't erase spend history.
